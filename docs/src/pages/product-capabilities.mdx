# Product Capabilities

This page organizes the documentation from a Product Owner and System Analyst
perspective. Start here when you want to understand what user outcome each part
of the library serves before diving into API details.

<h2 id="capability-map" class="toc-link">
  Capability Map
</h2>

| Capability              | User outcome                                                               | Documentation                                                            | Product spec                             |
| ----------------------- | -------------------------------------------------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------- |
| Dependency resolution   | Resolve dependencies by key, class, token, or constructor injection.       | [Container](/ts-ioc-container/container)                                 | `specs/epics/dependency-resolution.md`   |
| Scoped lifecycle        | Isolate application, request, transaction, page, or widget lifecycles.     | [Container](/ts-ioc-container/container)                                 | `specs/epics/scoped-lifecycle.md`        |
| Dependency registration | Describe classes, values, factories, keys, aliases, and scoped services.   | [Registration](/ts-ioc-container/registration)                           | `specs/epics/dependency-registration.md` |
| Provider behavior       | Cache, decorate, delay, restrict, or parameterize dependency creation.     | [Provider](/ts-ioc-container/provider), [Pipes](/ts-ioc-container/pipes) | `specs/epics/provider-behavior.md`       |
| Token-based injection   | Make dependency requests explicit, typed, reusable, and composable.        | [Token-Based Injection](/ts-ioc-container/token)                         | `specs/epics/token-based-injection.md`   |
| Injector strategies     | Support metadata, simple container, proxy, and custom injection styles.    | [Injector Strategies](/ts-ioc-container/injector)                        | `specs/epics/injector-strategies.md`     |
| Lifecycle hooks         | Run initialization, cleanup, property injection, and custom hook behavior. | [Lifecycle Hooks](/ts-ioc-container/hooks)                               | `specs/epics/lifecycle-hooks.md`         |
| Container modules       | Package container configuration by feature, environment, or lifecycle.     | [Registration](/ts-ioc-container/registration#container-modules)         | `specs/epics/container-modules.md`       |
| Metadata utilities      | Attach labels, tags, and reusable method behavior to application code.     | [Metadata Utilities](/ts-ioc-container/metadata)                         | `specs/epics/metadata-utilities.md`      |
| Errors and boundaries   | Make misconfiguration and unsupported usage diagnosable.                   | [Errors and Boundaries](/ts-ioc-container/errors-and-boundaries)         | `specs/epics/errors-and-boundaries.md`   |

<h2 id="reading-path" class="toc-link">
  Reading Path
</h2>

If you are evaluating the library, read in product order:

1. Dependency resolution and scoped lifecycle in [Container](/ts-ioc-container/container)
2. The [tsyringe alternative comparison](/ts-ioc-container/tsyringe-alternative)
3. Dependency registration in [Registration](/ts-ioc-container/registration)
4. Provider behavior in [Provider](/ts-ioc-container/provider) and [Pipes](/ts-ioc-container/pipes)
5. Token-based injection in [Token-Based Injection](/ts-ioc-container/token)
6. Injector strategies in [Injector Strategies](/ts-ioc-container/injector)
7. Lifecycle hooks in [Lifecycle Hooks](/ts-ioc-container/hooks)
8. Container modules in [Registration](/ts-ioc-container/registration#container-modules)
9. Metadata utilities in [Metadata Utilities](/ts-ioc-container/metadata)
10. Errors and boundaries in [Errors and Boundaries](/ts-ioc-container/errors-and-boundaries)

Framework guides such as React, SolidJS, Next.js, Express.js, and Fastify are
recipes that combine these core capabilities for a host environment.

<h2 id="traceability-model" class="toc-link">
  Traceability Model
</h2>

Every public behavior should be traceable from user need to executable test:

<CodeBlock
  code={`Business capability
  -> user story
  -> acceptance criteria
  -> executable spec test
  -> docs and ADRs`}
  lang="text"
/>

The product specs live in `specs/epics/`. Accepted product behavior is protected
by Vitest scenarios under `__tests__/specs/`. README examples under
`__tests__/readme/` remain executable usage documentation.

<h2 id="status" class="toc-link">
  Status
