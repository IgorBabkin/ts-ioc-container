# Spec-driven Workflow

The project uses specs to connect product intent, architecture decisions, tests,
and documentation.

<h2 id="artifact-map" class="toc-link">
  Artifact Map
</h2>

| Artifact               | Purpose                                                                                      |
| ---------------------- | -------------------------------------------------------------------------------------------- |
| `docs/adr/`            | Durable architecture and process decisions. ADRs explain why a direction was chosen.         |
| `specs/`               | Epics, stories, use cases, and acceptance criteria. Specs explain what behavior is promised. |
| `__tests__/specs/`     | Executable acceptance tests for public behavior.                                             |
| `__tests__/<feature>/` | Focused unit, integration, and regression tests for implementation details.                  |
| `__tests__/readme/`    | Executable examples included in generated documentation.                                     |
| `lib/`                 | Implementation of the accepted behavior.                                                     |

<h2 id="workflow" class="toc-link">
  Workflow
</h2>

<CodeBlock
  code={`Epic or use case
  -> behavior spec
  -> failing acceptance test
  -> implementation
  -> executable docs example
  -> generated README`}
  lang="text"
/>

Public behavior changes should start in `specs/`. The acceptance criteria then
become Vitest scenarios under `__tests__/specs/`. Implementation follows the
tests, and user-facing examples are updated only when the usage contract changes.

<h2 id="tdd-and-bdd" class="toc-link">
  TDD and BDD
</h2>

The repository uses BDD-style language for acceptance scenarios and a
TDD-compatible engineering loop.

- BDD gives the tests product language: what the user observes and why it
  matters.
- TDD gives the implementation loop: write the failing test, implement the
  smallest behavior, then refactor.

No separate Gherkin runner is required. Markdown specs plus Vitest acceptance
tests keep the process lightweight.

<h2 id="pull-request-checklist" class="toc-link">
  Pull Request Checklist
</h2>

Use this checklist when a change affects public behavior:

- Update or add the relevant spec in `specs/`.
- Add or update executable acceptance tests in `__tests__/specs/`.
- Update implementation tests in the feature folder when internals change.
- Update `__tests__/readme/`, docs pages, and `.readme.hbs.md` when examples or
  user-facing documentation change.
- Add an ADR when the change makes a durable architecture or process decision.
